# 第一部分：项目定位

## 1. 项目概述

Xenon 是一个 Rust 多维数组（张量）库，定位为科学计算场景下的数值基础设施。其角色类似于 Numpy 生态中的 ndarray 层，用于提供可供上层库复用的基础张量抽象、数据布局模型和核心运算能力。

### 1.1 目标用户

- 库开发者：稳定 API 与高性能互操作
- 系统开发者：在 `std` 环境下提供稳定的基础张量能力、底层内存控制、确定性数值行为以及最小依赖
- 间接用户：性能、正确性，以及与 Python 数值计算经验相一致的直觉行为

### 1.2 核心原则

- 正确性优先：优先保证类型安全、内存安全和数值正确性
- 性能敏感：保持高性能抽象，避免不必要的额外开销
- 清晰抽象：API 语义须明确，避免隐式行为
- 科学计算导向：保持与 BLAS 兼容的互操作预期
- 最小依赖：仅允许可选并行（`rayon`）与 SIMD 相关（`pulp`）外部依赖

### 1.3 工程约束

- Crate 结构：Xenon 须保持单 crate 结构，并遵循 SemVer
- 平台支持：Xenon 仅支持 `std` 环境

---

## 2. 项目范围

本节定义当前版本的产品范围。

### 2.1 范围内

- N 维数组的存储
- N 维数组的构造
- N 维数组的索引
- N 维数组的切片
- N 维数组的广播
- N 维数组的基础形状操作（当前版本仅支持 `transpose`）
- N 维数组的基础归约（当前版本仅支持 `sum`）
- N 维数组的基础集合操作（当前版本仅支持 `unique` 操作）
- N 维数组的逐元素运算
- N 维数组的显式类型转换
- 向量内积
- 供上游库零开销集成的原始指针 API
- 自定义复数类型
- 面向上游使用方的临时工作空间

### 2.2 范围外

- 矩阵-矩阵乘法、矩阵分解、对角化等高级线性代数能力
- 快速傅里叶变换、稀疏矩阵、自动微分和随机数能力
- BLAS/LAPACK 绑定（预期由上游库通过指针 API 自行集成）
- GPU 后端
- `serde` 序列化
- `arena` 分配器
- 栈分配小数组

# 第二部分：功能需求

## 3. 维度系统

维度系统用于界定张量 rank 与布局元数据的表示边界。

- 须支持 0 至 6 维静态维度类型，以及不受该范围限制的动态维度类型
- 须提供维度类型之间的转换能力
- 须提供从常见 Rust 类型构造维度类型的能力

---

## 4. 元素类型

- 基础操作的操作范围包括：张量构造、索引、切片、转置、广播、迭代遍历、等于与不等于比较、填充、连续化、逐元素类型转换、格式化输出、FFI 集成以及元数据查询
- 须支持以下张量元素类型：有符号整数（i32、i64）、浮点（f32、f64）、复数（Complex<f32>、Complex<f64>）以及 `bool`
- Xenon 须按运算能力对类型施加分层约束：

  | 能力层 | 适用类型 |
  |--------|----------|
  | 基础操作 | i32、i64、f32、f64、Complex<f32>、Complex<f64>、bool |
  | 四则运算 | i32、i64、f32、f64、Complex<f32>、Complex<f64> |
  | 数学函数 | f32、f64 |
  | 复数运算 | Complex<f32>、Complex<f64> |

- `bool` 作为张量元素类型时，仅支持基础操作（不含逐元素类型转换）与逻辑非
- `usize` 以及其它无符号整数类型在当前版本中仅承担元数据角色：仅可用于索引、轴、形状与切片边界，不属于张量元素类型，也不属于张量逐元素算术或逐元素类型转换的适用类型集合
- Xenon 不支持自动类型提升，任何类型转换均须显式进行
- 张量元素类型集合须为封闭集合，不支持下游扩展

---

## 5. 复数类型

- 须提供自定义复数类型，不依赖外部 crate
- 该复数类型须支持复数算术运算
- 该复数类型须支持与同精度实数之间的显式转换，具体转换语义遵循第 23 节。
- 跨精度运算也须通过显式转换完成。
- 涉及实数与复数的张量逐元素运算、运算符重载及相关 API 组合时，参与运算的双方元素类型须预先一致
- 该复数类型须支持格式化输出
- 其内存布局须采用稳定且可文档化的 FFI 表示；默认要求与 `#[repr(C)] struct { re: T, im: T }` 等价

---

## 6. 存储系统

### 6.1 存储模式

存储系统须显式区分所有权与可变性语义。

- 须提供多级别的数据访问控制与多种数据持有方式：
  - 只读：只读引用、共享只读引用
  - 可写：可写引用
  - 拥有：持有
- 只读引用指不拥有底层数据且以借用方式提供只读访问的存储模式
- 共享只读引用指可在多个张量实例之间共享同一底层数据但不提供可写访问权的存储模式
- 可写引用指不拥有底层数据但以独占借用方式提供可写访问的存储模式
- 持有指由张量实例拥有底层数据所有权的存储模式
- 当前版本不提供共享可写存储模式

### 6.2 存储模式转换

各存储模式之间的转换关系如下：

  | 源 → 目标 | 只读引用 | 共享只读引用 | 可写引用 | 持有 |
  |-----------|----------|-------------|----------|------|
  | 持有 | 零拷贝 | 零拷贝 | 零拷贝 | — |
  | 可写引用 | 零拷贝 | 零拷贝 | — | 须分配 |
  | 只读引用 | — | 不可转换 | 不可转换 | 须分配 |
  | 共享只读引用 | 零拷贝 | — | 不可转换 | 须分配 |

- 能够零拷贝完成的转换须直接支持
- 需要分配或生成独立持有结果的转换仅可在目标存储模式持有数据时执行
- 不满足前提的转换须返回可恢复错误
- 涉及零大小类型和空数组的操作不得引发未定义行为

**零拷贝**: 表示该转换不涉及数据复制，且其 API 语义须文档化为返回基于源张量重新解释或借出的结果，而非隐式复制；从可写引用零拷贝转换为共享只读引用时，结果须放弃可写访问权，并遵守源数据在该共享只读引用存续期间不得再被并发写入的独占性约束。

**须分配**: 表示该转换需要分配新存储并复制数据，且仅在目标为持有时方可执行。

**不可转换**: 表示该转换违反可变性、独占性约束或超出目标存储模式的可构造前提；对于已公开的运行时转换 API，此类情形须返回可恢复错误；仅当某类转换在 API 设计上被明确定义为不提供相应入口时，方可通过类型层拒绝。

---

## 7. 内存布局

内存布局用于约束拥有型布局、视图布局及其可观测性。

- 拥有型连续存储只支持列优先（F-order）布局
- 行优先（C-order）布局不在当前版本范围内
- 须支持对齐布局；允许存在仅用于对齐或实现目的的填充区域，但逻辑元素值及其访问结果须与不存在该填充区域时一致
- 张量视图可采用当前版本支持的合法 stride 布局，包括由转置产生的非连续布局，以及由广播产生的零步长布局
- 须提供布局状态查询能力，至少能够区分连续 F-order、非连续视图和广播视图等常见状态
- 当前版本不支持负步长布局

---

## 8. 张量类型

张量类型是面向用户的核心抽象，因此须提供充分的元数据查询能力与构造安全保证。

- 须提供以存储模式和维度类型为参数的张量类型
- 须提供常用张量类型的便捷别名
- 须提供形状、维度、步长、布局和数据位置等基础信息的查询能力
- 安全构造路径须验证一切可验证的合法性条件，至少包括：
  - shape 与 stride 组合在当前版本支持范围内且可表示
  - 元素总数计算不溢出
  - 逻辑访问范围不越过底层可见边界
  - 任一条件不满足时须返回可恢复错误
- 基于裸指针或其他外部不受控内存的构造路径须标记为不安全；其中仅验证可检查的元数据约束，其余前提由调用方保证

# 第三部分：计算需求

## 9. 计算加速

计算加速为可选能力，但不得改变对应公开 API 的外部语义契约。

### 9.1 SIMD

- 须提供可选的 SIMD 加速能力，默认关闭
- SIMD 路径须覆盖逐元素运算、归约和内积操作
- 当输入或环境不满足 SIMD 条件时，系统须自动回退到非 SIMD 路径
- SIMD 路径的精度约束如下：
  - 对整数和 `bool`，可观察结果须与标量路径逐元素一致
  - 对浮点和复数，须满足与标量路径相同的数值语义约束
  - 若存在已知舍入差异，须限制在第 28.3 节定义并文档化的误差容差内

### 9.2 并行

- 须提供可选的数据并行能力，默认关闭
- 须支持并行阈值配置
- 并行路径须支持逐元素运算、归约、内积操作
- 在已启用并行执行路径中，库内部不得再开启第二层并行

### 9.3 自动路径选择

- 系统须在相应能力已启用的前提下，根据数据规模和可用硬件能力自动选择 SIMD 或非 SIMD 路径，以及串行或并行执行路径
- 未启用的能力不得被自动选入执行路径
- 同一公开 API 在不同执行路径下须返回相同形状、相同错误类别，以及满足同一数值语义约束的结果

**数值语义约束**：

- 指第 28.3 节定义的 IEEE 754 行为约束、同执行路径下的确定性要求，以及文档化误差容差的适用边界
- 对浮点与复数类型，仅可在第 28.3 节允许的范围内存在已文档化的舍入容差；该容差范围内的差异不构成对数值语义约束的违反

---

## 10. 线程安全

- 张量类型的线程安全性须由其元素类型与底层存储模式共同决定
- 仅当相应前提成立时，方可在多线程之间安全传递或共享

---

## 11. 迭代器

迭代器行为须面向逻辑张量内容，而不受底层实现细节影响。

- 须支持按元素、按轴、按窗口、按索引遍历，以及多数组同步遍历
- 所有迭代器须提供精确的元素数量信息
- 空数组迭代须立即结束
- 带填充区域的数组迭代须仅遍历逻辑元素，不得暴露为对齐或实现目的引入的填充区域
- 不得对广播结果提供可变迭代能力
- 零维张量的元素迭代须恰好产出 1 个元素
- 按轴遍历和按窗口遍历不适用于零维张量，相关运行时 API 须通过可恢复错误拒绝此类请求

# 第四部分：API 需求

## 12. 逐元素运算

本节定义逐元素 API 的必备操作范围及其类型约束。须支持以下逐元素运算，并按类型进行分层约束：

  | 运算类别 | 具体运算 | 适用类型 |
  |----------|----------|----------|
  | 算术运算 | 加、减、乘、除 | i32、i64、f32、f64、Complex<f32>、Complex<f64> |
  | 一元运算 | 绝对值、符号函数 | i32、i64、f32、f64 |
  | 一元运算 | 取反、平方 | i32、i64、f32、f64、Complex<f32>、Complex<f64> |
  | 数学函数 | 正弦、平方根、指数、自然对数、下取整、上取整 | f32、f64 |
  | 复数运算 | 模、共轭 | Complex<f32>、Complex<f64>（模运算结果须为实数类型） |
  | 逻辑运算 | 逻辑非 | bool |
  | 比较运算 | 等于、不等于 | 全部张量元素类型 |
  | 比较运算 | 小于、大于 | i32、i64、f32、f64 |

- NaN 比较须遵循 IEEE 754
- 浮点有序比较采用 `PartialOrd` 语义，因此 NaN 与任何值（包括自身）的有序比较结果均为 `false`
- 须支持标量与张量之间的逐元素运算；标量须按可广播到目标张量全形状的零维输入语义处理
- 所有二元运算与比较运算均须支持广播
- 参与运算或比较的双方元素类型须预先一致；若输入形状按 NumPy 规则不兼容，须返回可恢复错误
- 有符号整数（i32、i64）的逐元素算术运算和平方运算中，凡发生溢出、除以零或结果不可表示（包括但不限于最小负值取绝对值、最小负值除以 `-1`）的情形，均须视为不可恢复错误，并遵循第 27 节定义的 panic 语义

## 13. 矩阵运算

- 当前版本仅支持向量内积
- 内积适用于以下元素类型：i32、i64、f32、f64、Complex<f32>、Complex<f64>
- `bool` 不支持内积
- 复数内积须满足共轭线性的数学定义；对于输入 `x` 和 `y`，结果须等价于 `sum(conj(x_i) * y_i)`
- 内积仅对逻辑上一维输入开放；若输入维度不为 1 或长度不匹配，须返回可恢复错误
- 有符号整数（i32、i64）内积的中间乘积或累加结果发生溢出时，须视为不可恢复错误，并遵循第 27 节定义的 panic 语义
- 空数组内积（两个长度为 0 的一维输入）须返回加法单位元（与第 14 节空数组 `sum` 归约语义一致）
  - `i32` / `i64` 返回 `0`
  - `f32` / `f64` 返回 `0.0`
  - `Complex<T>` 返回 `0 + 0i`

## 14. 归约

- 当前版本仅支持 `sum` 归约
- `sum` 归约适用于以下元素类型：i32、i64、f32、f64、Complex<f32>、Complex<f64>
- `bool` 不参与 `sum` 归约
- 须支持全局归约和沿指定轴归约；轴索引越界时须返回可恢复错误
- 归约须支持保留被归约轴为长度 1 的选项；当启用该选项时，输出 rank 须与输入一致，且被归约轴长度为 1
- 有符号整数归约发生溢出时，须视为不可恢复错误，并遵循第 27 节定义的 panic 语义
- 对沿指定轴的 `sum` 归约，若被归约轴长度为 `0`，则结果中每个输出位置均须返回对应元素类型的加法单位元；该规则同时适用于保留被归约轴与不保留被归约轴的两种形式
- 空数组 `sum` 归约须返回加法单位元：
  - `i32` / `i64` 返回 `0`
  - `f32` / `f64` 返回 `0.0`
  - `Complex<T>` 返回 `0 + 0i`

## 15. 集合操作

- 当前版本仅支持 `unique` 操作
- `unique` 须返回不重复元素；结果无需排序，且返回顺序不作要求
- `unique` 适用于 i32、i64、f32、f64、Complex<f32>、Complex<f64>
- `bool` 不支持`unique` 操作
- `unique` 的判定默认基于逐元素相等关系：
  - 对浮点，采用 Rust/IEEE 754 相等语义，因此 `NaN != NaN`、`-0.0 == 0.0`；输入中的每个 `NaN` 均须保留
  - 对复数，按分量分别比较并沿用对应实数语义

## 16. 广播

- 须遵循 NumPy 广播规则：从尾轴开始逐轴比对；当轴长度相同或其中一方为 1 时视为兼容，否则视为不兼容
- 广播操作不得拷贝数据
- 广播成功后结果须与源存储共享底层数据，且须作为共享只读引用对待（见第 6 节）
- 任何试图取得其可变访问权的 API 都须在类型层或运行时显式拒绝
- 须提供显式广播操作
- 当目标形状不兼容时须返回可恢复错误

## 17. 形状操作

- 当前版本仅支持转置操作
- 转置须返回共享底层数据的只读视图，并依据轴置换规则更新 shape 与 stride
- 该操作不得重排逻辑元素值
- 转置结果的存储模式须落在第 6 节定义的只读引用或共享只读引用范围内
- 即使源张量具备可写访问权，转置结果也不得暴露可写访问权

## 18. 索引操作

- 当前版本仅支持以 `usize` 表示的多维索引和范围索引（切片）；其类型角色遵循第 4 节对 `usize` 的统一约束
- 多维索引访问时，索引元组长度须与张量 rank 一致，且每个分量都须落在对应轴的有效范围内
- 对安全接口，任一条件不满足时须返回可恢复错误；不安全接口可将其中未检查前提交由调用方保证
- 由范围索引生成的切片须保持原有逻辑元素顺序，并依据起止边界更新 shape 与 stride
- 当切片与源张量共享底层数据时，其结果须遵循第 6 节约束，仅可落在只读引用或共享只读引用范围内，不提供共享可写视图

## 19. 构造操作

- 须提供以下标准构造方法，并按类型进行约束：
  - 全零：适用于 i32、i64、f32、f64、Complex<f32>、Complex<f64>、bool；其中 `bool` 对应 `false`
  - 全一：适用于 i32、i64、f32、f64、Complex<f32>、Complex<f64>、bool；其中 `bool` 对应 `true`
  - 单位矩阵：适用于 i32、i64、f32、f64、Complex<f32>、Complex<f64>，且仅对逻辑二维输入开放
- 须支持从动态数组、切片、固定数组和标量值构造张量
- 输入数据给出的顺序定义为逻辑索引语义上的元素对应关系，而非对底层连续内存线性排列的要求
- 对于由线性输入序列结合目标 shape 构造多维张量的情形，线性序列到逻辑多维索引的映射须与第 7 节规定的列优先（F-order）连续布局一致；构造结果须与该逻辑顺序一致
- 序列生成构造器（如 `arange`、`linspace` 等）不在当前版本范围内

## 20. 运算符重载

- 须支持四则运算的运算符语法，并支持广播
- 须支持张量与张量、张量与标量之间的运算；参与运算的双方元素类型须预先一致
- 所有组合均须产生新的独立张量
- 四则运算以外的运算符语法不在当前版本范围内

## 21. 实用操作

### 21.1 裁剪操作

- 须提供裁剪（clip）操作，用于将元素限制在 `[min, max]` 范围内
- 该操作不适用于复数类型与 `bool`
- `min` 与 `max` 须与张量元素类型一致
- 参数非法时：
  - 对整数，`min > max` 时返回可恢复错误
  - 对浮点，`min > max` 或任一边界为 `NaN` 时返回可恢复错误

### 21.2 填充操作

- 须提供填充（fill）操作，用于写入全部逻辑元素。
- 填充操作仅适用于提供可写访问权的张量
- 凡属于第 6 节定义的只读引用或共享只读引用存储模式的结果，均须拒绝填充请求
- 对于已公开的运行时填充 API，此类请求须返回可恢复错误
- 对带填充区域或非连续但可写的视图执行填充时，不得写入逻辑元素之外的内存区域

## 22. 连续性保证

- 须提供确保数据在内存中连续存储的操作；当输入已经是连续 F-order 存储时，该操作不得改变其逻辑值，并可复用现有数据
- 对于可正确访问和操作的非完全连续输入，连续化结果须为独立拥有型连续存储，且逻辑元素顺序须与连续化前一致

## 23. 类型转换

- 须提供逐元素类型转换方法
- 转换范围限于除 `bool` 外的张量元素类型集合（`i32`、`i64`、`f32`、`f64`、`Complex<f32>`、`Complex<f64>`）
- 须同时提供同类型拷贝接口
- `usize` 相关转换仅适用于第 4 节所定义的元数据角色，不属于逐元素类型转换范围
- 除本节显式列出的逐元素类型转换与同类型拷贝接口外，当前版本不再额外引入新的张量逐元素类型转换契约
- 如需提供与 Rust 核心库或标准库风格一致的转换接口，其语义与成功/失败条件也须完全受本节约束，不得扩大受支持的源/目标类型集合或改变默认错误语义

### 23.1 类型转换分类

  **无损转换（默认成功）：**

  | 源类型 | 目标类型 | 说明 |
  |--------|----------|------|
  | i32 | i64 | 整数宽化 |
  | f32 | f64 | 浮点宽化 |
  | i32 | f64 | 所有 i32 值可精确表示为 f64 |
  | i32 | Complex<f64> | 实部由 i32→f64 无损转换，虚部为 `0` |
  | f32 | Complex<f64> | 实部由 f32→f64 无损转换，虚部为 `0` |
  | f32 | Complex<f32> | 引入虚部 `0` |
  | f64 | Complex<f64> | 引入虚部 `0` |
  | Complex<f32> | f64 | 虚部为 `0` 时，实部由 f32→f64 无损转换 |
  | Complex<f32> | Complex<f64> | 分量宽化 |

  **有损转换（默认返回可恢复错误）：**

  | 源类型 | 目标类型 | 风险说明 |
  |--------|----------|----------|
  | i64 | i32 | 高位截断 |
  | f64 | f32 | 精度丢失 |
  | f32、f64 | i32、i64 | 小数部分截断 |
  | i32 | f32 | 部分值精度丢失（i32 精度范围超过 f32 尾数位数） |
  | i64 | f32、f64 | 部分值精度丢失（i64 精度范围超过 f64 尾数位数） |
  | i32 | Complex<f32> | 实部由 i32→f32 有损转换，虚部为 `0` |
  | i64 | Complex<f64> | 实部由 i64→f64 有损转换，虚部为 `0` |
  | i64 | Complex<f32> | 实部由 i64→f32 有损转换，虚部为 `0` |
  | f64 | Complex<f32> | 实部由 f64→f32 有损转换，虚部为 `0` |
  | Complex<f32> | f32 | 虚部丢弃；虚部为 `0` 时成功，否则返回可恢复错误 |
  | Complex<f32> | i32、i64 | 虚部为 `0` 时，按 f32→目标整数的规则处理；否则返回可恢复错误 |
  | Complex<f64> | f64 | 虚部丢弃；虚部为 `0` 时成功，否则返回可恢复错误 |
  | Complex<f64> | f32 | 虚部为 `0` 时，按 f64→f32 的规则处理；否则返回可恢复错误 |
  | Complex<f64> | i32、i64 | 虚部为 `0` 时，按 f64→目标整数的规则处理；否则返回可恢复错误 |
  | Complex<f64> | Complex<f32> | 分量精度丢失 |

  仅当文档明确标注时，方可另行提供截断、饱和等替代接口

### 23.2 类型转换规则

除同类型拷贝外，凡属于上述受支持元素类型集合、但未在表中逐项单列的转换组合，其语义须按以下闭合规则确定：
- 实数到复数：先按对应实数类型到目标复数实部分量类型的规则转换实部，再引入值为 `0` 的虚部
- 复数到实数：仅当虚部为 `0` 时方可继续转换；随后按其实部到目标实数类型的规则处理；若虚部不为 `0`，须返回可恢复错误
- 复数到复数：按实部与虚部分量分别应用对应实数类型之间的转换规则
- 由上述规则组合后，只要任一步属于无损转换，则该步默认成功；任一步属于有损转换，则整体默认返回可恢复错误

## 24. 格式化输出

- 面向用户的格式化输出须简洁且可读
- 同一张量在相同截断阈值和相同格式化配置下，其输出须稳定一致
- 数据展示须遵循 NumPy 风格，至少包括：以嵌套括号表示维度层级、按逻辑顺序展示元素，以及以可区分方式显示标量与零维张量
- 当数组元素数量超过指定阈值时，输出须触发截断
- 截断结果中须保留足以识别 shape 及前后局部数据的可读信息

## 25. FFI 集成

- 须提供原始指针 API，供上游库以零开销方式导出或接入底层表示
- “零开销”仅指 Xenon 对底层指针与元数据的暴露本身不额外引入数据复制或隐式重排，不意味着当前版本支持的所有合法布局都天然满足外部库（如 BLAS/LAPACK）的直接调用前提
- 须支持从裸指针构造张量，以及将张量导出为裸指针表示；前者须提供明确的不安全接口，调用方须满足以下前提条件：
  - 指针有效性：指针非悬垂
  - 对齐：满足元素类型要求
  - 可访问范围：覆盖逻辑访问区域
  - 生命周期：覆盖张量使用期
- 须提供多维索引到指针和偏移量的转换能力；对合法索引计算出的地址，须与按 shape/stride 推导得到的逻辑位置一致
- 须提供面向上游 BLAS/LAPACK 集成的辅助 API，包括对满足相应前提的数据提供列优先布局保证、leading dimension 查询，以及面向当前版本合法 stride 的指针 + stride 元数据导出格式；该要求旨在支持上游完成兼容性判断与适配，但不包含实际 BLAS 例程调用

## 26. 临时工作空间

- 须提供支持对齐分配的临时工作空间类型
- 该工作空间须支持分割与动态扩容
- 分割后的各子工作空间在逻辑上须互不重叠
- 扩容操作仅可在不存在活跃数据借用时执行，且须通过独占访问保证引用安全
- 扩容后不得继续使用扩容前导出的内部指针或视图

# 第五部分：工程保障

## 27. 错误处理

- 可恢复错误须以返回值形式报告
- 不可恢复错误须通过 panic 终止当前执行路径；在 `panic = abort` 等构建环境下可表现为进程终止
- 凡各章节明确规定为不可恢复的算术域错误，包括有符号整数溢出、整数除以零及结果不可表示等情形，均须遵循统一的 panic 语义
- 所有可恢复错误，以及不可恢复错误对应的 panic 信息，须携带足以定位问题的诊断信息，至少包括错误类别，以及触发错误的轴、shape、索引、目标类型等相关上下文中的适用子集
- 不可恢复错误发生后，系统须最小化资源泄漏风险；在 `panic = abort` 等无法执行展开清理的构建环境下，允许进程级终止导致资源不回收
- 并行操作中发生不可恢复错误时，错误须立即传播，不得静默忽略
- 所有资源释放逻辑均不得引发不可恢复错误

## 28. 质量要求

### 28.1 文档

- 所有公开 API 须提供文档说明
- 所有标记为不安全的函数须提供安全性说明
- 关键 API 须提供使用示例

### 28.2 测试

- 须包含单元测试、集成测试、边界测试，以及针对 feature gate 和类型边界的编译期/配置测试

### 28.3 数值精度

- 基础浮点算术、比较以及 NaN/Inf 相关行为须满足 IEEE 754 标准
- 基础张量操作与基础数值运算须在同平台、同编译配置、同执行路径下结果确定
- 跨平台行为须满足 IEEE 754 语义约束，但不承诺 bitwise 一致
- 对浮点与复数结果，仅当不同执行路径引入已知舍入差异时，方可使用文档化误差容差进行约束
- 仅在 `std` 环境下提供的数学函数也须保证语义一致，并通过文档化的误差容差进行约束
- 文档须明确说明各类误差容差的适用范围

**基础张量操作**： 至少包括逐元素运算、广播、转置、索引/切片、归约、内积、连续化与类型转换

**基础数值运算**： 至少包括第 12 节和第 13 节中定义的标量与张量数值计算行为

### 28.4 边界覆盖

- 须覆盖以下边界情况：

  | 边界类别 | 示例 |
  |----------|------|
  | 空张量 | 零元素数组 |
  | 单元素 | 含恰好 1 个元素的张量 |
  | 大张量 | 元素数量达到 `10^7` 量级或总字节数达到 GiB 量级 |
  | 极端值 | 类型边界值（如最大/最小值、NaN、Inf） |
  | 非连续布局 | 由转置等操作产生的非连续视图 |
  | 高维 | 接近静态维度上限的张量，以及高 rank 的动态维度张量 |
  | 非法元素类型 | 违反第 4 节元素类型约束的输入 |

### 28.5 并行与 SIMD 测试

- 并行和 SIMD 相关功能须保证无数据竞争和未定义行为
- 并行归约结果须在相同输入、相同执行路径和相同配置下与对应串行实现保持语义一致
- 若对浮点或复数结果允许存在已文档化的舍入差异，须明确说明其适用范围
- 须验证 `usize` 与其它无符号整数满足第 4 节定义的约束
