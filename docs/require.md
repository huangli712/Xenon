版本: v23
日期: 2026-04-12
许可: MIT
MSRV: Rust 1.85+

# 第一部分：项目定位

## 1. 项目概述

Xenon 是一个 Rust 多维数组（张量）库，定位为科学计算的数值基础设施，类似于 Python 生态中 NumPy ndarray 层的多维数组基础设施。

### 1.1 目标用户

| 用户类型 | 核心诉求 |
|----------|----------|
| 库开发者 | 稳定 API、高性能互操作 |
| 系统开发者 | 基础张量能力的 `no_std + alloc` 支持、底层内存控制、确定性数值行为、最小依赖 |
| 间接用户 | 性能、正确性、与 Python 经验的直觉一致性 |

### 1.2 核心原则

| 原则 | 说明 |
|------|------|
| 正确性优先 | 类型安全、内存安全、数值精度 |
| 性能敏感 | 高性能抽象 |
| 清晰抽象 | API 语义明确，无隐式行为 |
| 科学计算导向 | BLAS 兼容 |
| 最小依赖 | 仅允许可选的并行（rayon）和 SIMD 相关（pulp）外部依赖 |

### 1.3 工程约束

| 约束 | 要求 |
|------|------|
| Crate 结构 | 单 crate，遵循 SemVer |
| no_std 支持 | 支持 `no_std + alloc` 的基础张量能力；`std` 为默认，并提供仅在 `std` 下可用的数学函数能力 |

---

## 2. 项目范围

### 2.1 范围内

- N维数组的存储、构造
- N维数组的索引操作、切片操作、形状操作、归约操作、广播操作、集合操作
- N维数组的逐元素运算
- N维数组的显式类型转换
- 向量内积
- 原始指针 API，供上游库以零开销方式集成
- 自定义复数类型
- 临时工作空间（供上游库使用）

### 2.2 范围外

- 矩阵-矩阵乘法、矩阵分解、对角化等高级线性代数
- 快速傅里叶变换、稀疏矩阵、自动微分、随机数
- BLAS/LAPACK 绑定（由上游库通过指针 API 集成）
- GPU 后端
- serde 序列化
- arena 分配器
- 栈分配小数组

# 第二部分：功能需求

## 3. 维度系统

- 须支持 0 至 6 维静态维度类型，以及不受该范围限制的动态维度类型
- 须提供维度类型之间的转换
- 须提供从常见 Rust 类型构造维度类型的能力
- 须支持负步长

---

## 4. 元素类型

- 须支持以下元素类型：有符号整数（i32、i64）、浮点（f32、f64）、复数（Complex<f32>、Complex<f64>）、bool
- `usize` 仅用于索引、轴和形状元数据，不属于张量元素类型
- 须按运算能力对类型进行分层约束：基础操作（全部张量元素类型）、四则运算（i32、i64、f32、f64、Complex<f32>、Complex<f64>）、数学函数（仅 f32、f64，且仅在 `std` 下可用）、复数运算（仅 Complex<f32>、Complex<f64>）
- 不支持自动类型提升；类型转换须显式进行
- 元素类型集合须为封闭集合，不支持下游扩展

---

## 5. 复数类型

- 须提供自定义复数类型，不依赖外部 crate
- 须支持复数算术运算
- 须支持复数与同精度实数的混合运算，跨精度须显式转换
- 须支持格式化输出
- 内存布局须采用稳定、可文档化的 FFI 表示；默认要求与 `#[repr(C)] struct { re: T, im: T }` 等价
- 若需声明与 C99 `_Complex` ABI 兼容，须将兼容范围限制到经过验证的目标平台/ABI，并在文档中明确列出

---

## 6. 存储系统

- 须提供多级别的数据访问控制（只读、可写、拥有）
- 须支持数据的多种持有与访问方式（持有、只读引用、可写引用、共享只读引用）
- 涉及零大小类型和空数组的操作不得引发未定义行为

---

## 7. 内存布局

- 拥有型连续存储只支持列优先（F-order）布局
- 行优先（C-order）布局不在当前范围内
- 须支持对齐布局；其中允许存在仅服务于对齐或实现目的的填充区域，但逻辑元素值及其访问结果须与无该填充区域时一致
- 张量视图可采用任意合法 stride 布局，包括转置产生的非连续布局、负步长布局及广播产生的零步长布局
- 须提供布局状态查询能力，至少可区分连续 F-order、非连续视图、广播视图等常见状态

---

## 8. 张量类型

- 须提供以存储模式和维度类型为参数的张量类型
- 须提供常用张量类型的便捷别名
- 须提供形状、维度、步长、布局、数据位置等基础信息的查询能力
- 安全构造路径须验证一切可验证的合法性条件，至少包括：shape 与 stride 组合可表示、元素总数计算不溢出、逻辑访问范围不越过底层可见边界；任一条件不满足时须返回可恢复错误
- 基于裸指针或其他外部不受控内存的构造路径须标记为不安全；其中仅验证可检查的元数据约束，其余前提由调用方保证

---

# 第三部分：计算需求

## 9. 计算加速

### 9.1 SIMD

- 须提供可选的 SIMD 加速功能，默认关闭
- 须为逐元素运算、归约、内积等操作提供加速路径
- 不满足 SIMD 条件时须自动回退
- 对整数与 `bool`，SIMD 路径的可观察结果须与标量路径逐元素一致；对浮点与复数，须满足与标量路径相同的语义约束，若存在已知舍入差异，须限定在文档化误差容差内

### 9.2 并行

- 须提供可选的数据并行功能，默认关闭
- 须支持并行阈值配置
- 须支持逐元素运算、归约、函数映射、多数组同步操作的并行执行
- 禁止库内部在已启用的并行执行路径中再次开启第二层并行

### 9.3 自动路径选择

- 在并行功能启用后，须根据数据规模和可用硬件能力自动选择串行或并行执行路径；同一公开 API 在两条路径下须返回相同形状、相同错误类别，以及满足同一数值语义约束的结果

---

## 10. 线程安全

- 张量类型的线程安全性须由其元素类型与底层存储模式共同决定；仅当相应前提满足时，方可在多线程之间安全传递或共享

---

## 11. 迭代器

- 须支持按元素、按轴、按窗口、按索引遍历，以及多数组同步遍历
- 所有迭代器须提供精确的元素数量信息
- 空数组迭代须立即结束
- 带填充区域的数组迭代须仅遍历逻辑元素，不得暴露为对齐或实现目的引入的填充区域
- 禁止对广播结果进行可变迭代
- 零维张量元素迭代产出恰好 1 个元素，不支持按轴和按窗口遍历；相关运行时 API 须通过可恢复错误拒绝此类请求

---

# 第四部分：API 需求

## 12. 逐元素运算

- 须支持算术运算（加、减、乘、除），适用于数值元素类型（i32、i64、f32、f64、Complex<f32>、Complex<f64>）
- 须支持一元运算，并按类型分层约束：绝对值（i32、i64、f32、f64）、取反（i32、i64、f32、f64、Complex<f32>、Complex<f64>）、符号函数（i32、i64、f32、f64）、平方（i32、i64、f32、f64、Complex<f32>、Complex<f64>）
- 须支持数学函数（仅 f32、f64，且仅在 `std` 下可用）：正弦、平方根、指数、自然对数、下取整、上取整
- 须支持复数运算（模、共轭），模运算结果须为实数类型
- 须支持逻辑非（仅 bool）
- 须支持比较运算，并按类型分层约束：等于、不等于（全部张量元素类型），小于、大于（仅 i32、i64、f32、f64），返回 bool 类型张量
- NaN 比较遵循 IEEE 754；浮点有序比较采用 `PartialOrd` 语义，因此 NaN 与任何值（包括自身）的有序比较结果均为 false
- 须支持标量与张量的逐元素运算；标量应按可广播到目标张量全形状的零维输入语义处理
- 所有二元运算和比较运算须支持广播；若输入形状按 NumPy 规则不兼容，须返回可恢复错误

## 13. 矩阵运算

- 当前版本只支持向量内积
- 复数内积须满足共轭线性的数学定义；对输入 `x`、`y`，结果须等价于 `sum(conj(x_i) * y_i)`
- 内积仅对逻辑上一维输入开放；维度不是 1 或长度不匹配时须返回可恢复错误

## 14. 归约

- 当前版本只支持 `sum` 归约
- 须支持全局归约和沿指定轴归约；轴索引越界时须返回可恢复错误
- 归约须支持保留被归约轴为长度 1 的选项；当启用该选项时，输出 rank 须与输入一致，且被归约轴长度为 1
- 有符号整数归约溢出时视为不可恢复错误，并遵循第 27 节定义的 panic 语义
- 空数组 sum 归约返回加法单位元：`i32/i64 -> 0`，`f32/f64 -> 0.0`，`Complex<T> -> 0 + 0i`
- `bool` 不参与 `sum` 归约

## 15. 集合操作

- 当前版本只支持唯一值操作，返回不重复元素（无需排序，且返回顺序不作要求）
- 唯一值操作须对 i32、i64、f32、f64、Complex<f32>、Complex<f64> 可用
- 唯一值操作不适用于 `bool`
- 唯一值判定默认基于逐元素相等关系：对浮点遵循 Rust/IEEE 754 相等语义，因此 `NaN != NaN`，`-0.0 == 0.0`；复数按分量分别比较并沿用对应实数语义

## 16. 广播

- 须遵循 NumPy 广播规则：从尾轴开始逐轴比对，轴长度相同或其中一方为 1 时兼容，否则不兼容
- 广播操作不得拷贝数据；广播成功后结果须与源存储共享底层数据
- 广播结果须为只读；任何试图取得其可变访问权的 API 都须在类型层或运行时显式拒绝
- 须提供显式广播方法；当目标形状不兼容时须返回可恢复错误

## 17. 形状操作

- 当前版本只提供转置与 reshape 操作
- 转置须返回共享底层数据的视图，并按轴置换规则更新 shape 与 stride，不得重排逻辑元素值
- reshape 仅当目标 shape 的元素总数与输入逻辑元素总数相等时方可成功；否则须返回可恢复错误

## 18. 索引操作

- 当前版本只支持以 `usize` 表示的多维索引，以及以 `usize` 边界表示的范围索引（切片）
- 多维索引访问时，索引元组长度须与张量 rank 一致，且每个分量都落在对应轴的有效范围内；否则须返回可恢复错误或在不安全接口外显式拒绝
- 范围索引生成的切片须保持原有逻辑元素顺序，并依据切片参数更新 shape 与 stride

## 19. 构造操作

- 须为支持相应构造语义的张量元素类型提供全零、全一、单位矩阵等标准构造方法；其中单位矩阵仅对逻辑二维且支持 `0/1` 单位元语义的数值类型开放
- 须支持从动态数组、切片、固定数组、标量值构造张量；构造结果的逻辑元素排列须与输入数据给出的顺序一致
- 当前版本不提供序列构造器

## 20. 运算符重载

- 须支持四则运算的运算符语法，并支持广播
- 须支持张量与张量、张量与标量之间的运算
- 所有组合均产生新的独立张量
- 四则运算以外的运算符语法不在当前范围内

## 21. 实用操作

- 须提供裁剪操作：将元素限制在 `[min, max]` 范围；该操作不适用于复数类型与 `bool`，且 `min` 与 `max` 须与张量元素类型一致
- 对整数类型，`min > max` 须作为可恢复错误；对浮点类型，若 `min > max` 或任一边界为 `NaN`，须作为可恢复错误
- 须提供填充操作：填充所有逻辑元素；对带填充区域或非连续视图执行填充时，不得写入逻辑元素之外的内存区域

## 22. 连续性保证

- 须提供确保数据在内存中连续存储的操作；当输入已经是连续 F-order 存储时，该操作不得改变其逻辑值，并可复用现有数据语义
- 须同时支持数据在内存中非完全连续但仍可正确访问和操作的场景；对这类输入执行连续化后，结果须为独立拥有型连续存储，且逻辑元素顺序与连续化前一致

## 23. 类型转换

- 须提供逐元素类型转换方法，转换范围限于支持的张量元素类型集合
- 浮点到整数、宽整数到窄整数，以及实数/复数间可能丢失信息的转换，须提供显式的错误语义：默认须返回可恢复错误；仅在文档明确标注时，方可另行提供截断或饱和等替代接口
- 须同时提供同类型拷贝接口
- 类型转换仅适用于持有数据的存储模式
- 须支持各存储模式之间的转换
- `usize` 相关转换仅适用于索引、轴和形状元数据，不属于逐元素类型转换范围
- 张量类型须支持 Rust 核心/标准转换接口中与编译环境兼容的类型转换约定

## 24. 格式化输出

- 面向用户的格式化输出须简洁可读；同一张量在相同截断阈值和相同格式化配置下输出须稳定一致
- 数据展示须遵循 NumPy 风格，至少包括：嵌套括号表示维度层级、元素按逻辑顺序展示、标量与零维张量采用可区分的显示形式
- 超过指定元素数量的数组须触发输出截断；截断输出中须保留足以识别 shape 与前后局部数据的可读信息

## 25. FFI 集成

- 须提供原始指针 API，供上游库以零开销方式集成
- 须支持从裸指针构造张量和将张量导出为裸指针表示；前者须提供明确的不安全接口，并文档化调用方需保证的前提条件，至少包括：指针非悬垂、对齐满足元素类型要求、可访问范围覆盖逻辑访问区域、生命周期覆盖张量使用期
- 须提供多维索引到指针和偏移量的转换；对合法索引计算出的地址须与按 shape/stride 推导的逻辑位置一致
- 须提供 BLAS 兼容性相关 API（指列优先布局保证、leading dimension 查询、指针 + stride 传递格式等，不含实际 BLAS 例程调用）

## 26. 临时工作空间

- 须提供临时工作空间类型，支持对齐分配
- 须支持 `no_std` 环境
- 须支持工作空间的分割与动态扩容；分割后的各子工作空间逻辑上须互不重叠
- 扩容操作仅可在无活跃数据借用时执行，须通过独占访问保证引用安全；扩容后不得继续使用扩容前导出的内部指针或视图

# 第五部分：工程保障

## 27. 错误处理

- 可恢复错误须以返回值形式报告
- 不可恢复错误须通过 panic 终止当前执行路径；在 `panic = abort` 等构建环境下可表现为进程终止
- 所有可恢复错误，以及不可恢复错误对应的 panic 信息，须携带足以定位问题的诊断信息，至少包括错误类别，以及触发错误的轴/shape/索引/目标类型等相关上下文中的适用子集
- 不可恢复错误发生后须最小化资源泄漏风险；在 `panic = abort` 等无法执行展开清理的构建环境下，允许进程级终止导致资源不回收
- 并行操作中发生不可恢复错误时须立即传播，不得静默忽略
- 所有资源释放逻辑不得引发不可恢复错误

## 28. 质量要求

### 28.1 文档

- 所有公开 API 须提供文档说明
- 所有标记为不安全的函数须提供安全性说明
- 关键 API 须提供使用示例；仅在 `std` 下可用的数学函数示例须明确标注其构建前提

### 28.2 测试

- 须包含单元测试、集成测试、边界测试，以及针对 feature gate 和类型边界的编译期/配置测试

### 28.3 数值精度

- 基础浮点算术、比较和 NaN/Inf 相关行为须满足 IEEE 754 标准
- 须保证基础张量操作与基础数值运算在同平台、同编译配置、同执行路径下结果确定；跨平台行为须满足 IEEE 754 语义约束，但不承诺 bitwise 一致。仅在 `std` 下提供的数学函数须保证语义一致，并通过文档化的误差容差进行约束

### 28.4 边界覆盖

- 须覆盖以下边界情况：空张量、单元素、大张量（如元素数量达到 `10^7` 量级或总字节数达到 GiB 量级）、极端值、非连续布局、高维、`std`/`no_std` 能力差异、非法元素类型边界（如 `usize` 不得作为张量元素类型）等

### 28.5 并行与 SIMD 测试

- 并行和 SIMD 相关功能须保证无数据竞争和未定义行为
- 并行归约结果须在相同输入、相同执行路径和相同配置下与对应串行实现保持语义一致；对浮点结果允许存在已文档化的舍入差异时，须明确说明适用范围
- 须验证数学函数在 `std` 构建下可用、在 `no_std` 构建下不可用；须验证 `usize` 不满足张量元素类型与算术类型约束
