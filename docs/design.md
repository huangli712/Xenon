# 设计方案文档规范

> 本文档定义 Xenon 项目设计方案文档的编写规范。
> 它是面向 `./design/` 下设计文档的元规范，本身不是某个模块的设计方案文档。
> `./design/` 下的文档分为两类：模块设计文档与横切规范文档。
> 两类文档均须在 `./require.md` 的当前版本范围内遵循本规范，以确保文档的一致性、可读性和可执行性。

---

## 1. 文件命名与编号

### 1.1 扁平全局编号

采用 `NN-模块名.md` 的扁平编号格式，NN 为两位全局递增序号：

```
00-rust-standards.md
01-architecture.md
02-dimension.md
03-element.md
...
```

**优点**：引用时只需一个数字（"参见文档 10"），跨文档交叉引用简单。

### 1.2 文档分组通过目录或编号段体现

若模块数量较多，可通过编号段自然分组：

| 编号段 | 含义 |
|--------|------|
| `00-01` | 全局规范与架构概览 |
| `02-07` | 核心类型系统（维度、元素、复数、存储、布局、张量） |
| `08-09` | 计算后端（SIMD、并行） |
| `10-23` | API 模块（迭代器、运算、归约、集合、广播、形状、索引、构造、运算符重载、实用操作、类型转换、格式化输出、FFI） |
| `24` | 临时工作空间 |
| `25-26` | 横切关注点（线程安全、错误处理） |
| `27-29` | 质量保障（基准测试、集成测试、文档） |

### 1.3 文件位置声明

模块设计文档第 3 节必须声明文件位置；横切规范文档若不对应单一源码位置，可改为“适用范围/影响范围”声明：

````markdown
## 3. 文件位置

```
src/
├── dimension.rs       # 所有维度类型和 Dimension trait
```

单文件设计：维度类型之间高度相关，拆分反而增加耦合复杂度。
````

## 2. 文档元信息头

每个文档开头必须包含元信息块：

```markdown
# <文档标题>

> 文档编号: NN | 模块: `src/<module>/` | 阶段: Phase X
> 前置文档: `00-rust-standards.md`, `01-architecture.md`, ...
> 需求参考: 需求说明书 §X, §Y, §Z
> 范围声明: 范围内 / 范围外
```

**前置文档链**：明确列出阅读本文档前须先理解的文档，形成阅读顺序指引。

其中：
- 模块设计文档可采用 `# <模块名>模块设计`
- 横切规范文档可采用 `# <主题名>规范`、`# <主题名>设计约束` 等不带模块语义的标题
- 模块设计文档中的 `模块: ` 字段填写对应源码路径
- 横切规范文档中的该字段可替换为 `适用范围:` 或 `影响范围:`

## 3. 标准章节结构

`./design/` 下的文档分为以下两类：

- **模块设计文档**：面向单个模块、类型簇或 API 族的实施方案文档
- **横切规范文档**：面向线程安全、错误处理、质量要求等跨多个模块的约束性文档

### 3.1 模块设计文档

每个**模块**实施方案文档必须包含以下章节（按需增删）。

> **豁免**：全局规范文档（如 `00-coding.md`、`01-architecture.md`）以及横切规范文档不是模块文档，可不遵循以下标准章节结构，采用适合其内容的结构。

| 章节 | 必须 | 说明 | 规范章节 |
|------|------|------|---------|
| 1. 模块定位/概述 | ✅ | 职责、包含/不包含、设计原则 | §4 |
| 2. 需求映射与范围约束 | ✅ | 对应需求条款、范围内、范围外、非目标 | §5 |
| 3. 文件位置 | ✅ | 文件路径 + 划分理由 | §1.3 |
| 4. 依赖关系 | ✅ | 依赖图 + 具体消费的类型/trait + 依赖合法性声明 | §6 |
| 5. 公共 API 设计 | ✅ | 公共接口草案、关键签名、文档/示例交付物 | §7 |
| 6. 内部实现设计 | ✅ | 算法、数据结构、安全性论证 | §8 |
| 7. 实现任务拆分 | ✅ | Wave/Phase 分组，统一粒度 | §9 |
| 8. 测试计划 | ✅ | 单元 + 集成 + 边界 + feature gate / 编译期配置 + 类型边界；属性测试按需补充 | §10 |
| 9. 模块交互设计 | ✅ | 接口约定、数据流 | §11 |
| 10. 错误处理与语义边界 | ✅ | recoverable error、panic、诊断字段、路径一致性 | §12 |
| 11. 设计决策记录 | 推荐 | ADR 格式 | §13 |
| 12. 性能描述 | 按需 | 复杂度、内存、缓存行为 | §14 |
| 13. 平台与工程约束 | ✅ | std-only、单 crate、SemVer、依赖约束 | §15 |
| 附录 | 按需 | API 参考、测试清单、运行命令 | — |

### 3.2 横切规范文档

横切规范文档不强制套用模块实施方案模板，但须至少覆盖以下内容：

| 章节 | 必须 | 说明 |
|------|------|------|
| 1. 主题定位/适用范围 | ✅ | 说明约束主题、覆盖范围、适用对象 |
| 2. 需求映射与范围约束 | ✅ | 对应需求条款、范围内、范围外、非目标 |
| 3. 影响范围或涉及模块 | ✅ | 说明受该规范影响的模块、路径或能力集合 |
| 4. 规范内容/约束定义 | ✅ | 明确规则、边界、禁止项、兼容性要求 |
| 5. 验证与落地方式 | ✅ | 测试、检查项、执行路径一致性或评审要求 |
| 6. 平台与工程约束 | ✅ | std-only、单 crate、SemVer、依赖约束 |
| 附录 | 按需 | 术语、示例、检查清单、运行命令 |

横切规范文档仅在确有必要时才包含“文件位置”“公共 API 设计”“内部实现设计”“模块交互设计”等模块化章节。

> **示例约束**：本文档中的示例、伪代码、模块名和 API 片段仅用于说明写法，不得扩展 `docs/require.md` 当前版本范围。
> 若示例涉及范围外能力，必须明确标注为“反例”或替换为范围内能力示例。

## 4. 模块定位与职责边界

### 4.1 职责边界表（必须）

每个模块开篇必须用"包含/不包含"表明确职责边界：

```markdown
| 职责 | 包含 | 不包含 |
|------|------|--------|
| 元素遍历 | 按逻辑元素顺序遍历 | 具体运算逻辑 |
| 轴遍历 | 沿指定轴产出子视图或子结果 | 子结果的具体操作 |
| 索引遍历 | 产出合法多维索引或索引相关访问能力 | 新的索引语法或越界宽松语义 |
```

### 4.2 设计原则

用简洁的表格列出模块遵循的设计原则：

```markdown
| 原则 | 体现 |
|------|------|
| 步长感知 | 正确处理连续、转置后非连续与广播等合法布局 |
| 零拷贝 | 索引/切片/转置/广播等视图类结果优先复用底层数据 |
| trait 一致性 | 仅在需求要求的遍历能力上提供一致且可验证的接口语义 |
```

## 5. 需求映射与范围约束

### 5.1 需求映射与范围约束（必须）

每个模块设计文档必须显式说明其承接的需求条款与边界，避免方案越出 `require.md` 的当前版本范围：

```markdown
| 类型 | 内容 |
|------|------|
| 需求映射 | `需求说明书 §6.1`, `§8`, `§18`, `§28.2` |
| 范围内 | 只读/可写视图的索引规则与错误返回 |
| 范围外 | 负步长、共享可写视图、GPU 后端 |
| 非目标 | 不在本设计中引入新的存储模式或额外第三方依赖 |
```

若需求存在范围外条目（如 BLAS 绑定、GPU、`serde`、`arena` 等），文档须明确声明“不在本文覆盖范围内”，不得默认留白。

## 6. 依赖关系描述

### 6.1 依赖图（ASCII）

使用 ASCII 图展示模块间的依赖方向：

```
src/iter/
├── crate::tensor        # TensorBase<S, D>, 类型别名
├── crate::dimension     # Dimension trait, Ix0~Ix6, IxDyn
├── crate::storage       # Storage, RawStorage trait
└── crate::broadcast     # broadcast_shape() and compatibility checks
```

### 6.2 依赖精确到类型级（必须）

不仅列出依赖哪些模块，还须列出**具体消费了哪些类型/trait**：

```markdown
| 来源模块 | 使用的类型/trait |
|----------|-----------------|
| `tensor` | `TensorBase<S, D>`, `TensorView`, `.shape()`, `.as_ptr()` |
| `dimension` | `Dimension`, `Ix0`~`Ix6`, `IxDyn` |
| `layout` | `LayoutFlags`, `Order` |
```

### 6.3 依赖方向声明

显式声明依赖方向：

> **依赖方向：单向向上。** `iter/` 仅消费 `tensor` 和 `dimension` 等核心模块，不被它们依赖。

### 6.4 依赖合法性与新增依赖说明（必须）

除列出依赖关系外，每个模块设计文档还须声明新增依赖是否合法：

```markdown
| 项目 | 说明 |
|------|------|
| 新增第三方依赖 | 无 / `rayon` / `pulp` / 其他（须说明原因） |
| 合法性结论 | 是否符合需求说明书对最小依赖的限制 |
| 替代方案 | 若不合法，说明为何不能采用既有模块或标准库实现 |
```

若设计涉及自定义复数类型、FFI 表示或 feature gate，须额外说明其方案是否引入需求基线禁止的外部 crate 或多 crate 拆分。

## 7. 公共 API 设计

### 7.1 公共接口草案与关键签名

API 设计部分至少应包含：
- 对外暴露的核心类型、trait 或函数
- 关键方法签名与必要的泛型约束
- 不安全接口的 `# Safety` 要点
- 关键 API 的文档说明与使用示例交付计划

### 7.2 Good / Bad 对比式代码示例（必须）

关键 API 的使用方式须配有 `Good` / `Bad` 对比示例：

```rust
// Good - use `?` and `Result` for shape incompatibility
pub fn broadcast_to<D2>(&self, shape: D2) -> Result<TensorView<'_, A, D2>> {
    if !is_broadcast_compatible(self.shape(), shape.as_ref()) {
        return Err(XenonError::IncompatibleShape);
    }
    // ...
}

// Bad - hide shape incompatibility as a panic
pub fn broadcast_to_bad<D2>(&self, shape: D2) -> TensorView<'_, A, D2> {
    assert!(is_broadcast_compatible(self.shape(), shape.as_ref()));
    // ...
}
```

示例代码全部使用英文（包括代码与注释）。

### 7.3 设计决策内联标注

关键设计选择用引用块标注：

```markdown
> **设计决策：** `AxisIter` 的 `Item` 类型为 `TensorView<'a, A, D::Smaller>`。
> 这要求 `D` 满足 `RemoveAxis`，并由 `RemoveAxis` trait 提供 `type Smaller: Dimension` 关联类型。
```

## 8. 内部实现设计

### 8.1 算法描述

核心算法用伪代码或步骤列表描述：

```
//
// the following pseudo-codes are used to illustrate the core algorithm.
//
increment_index_f(shape, index):
    for i in 0..ndim:
        index[i] += 1
        if index[i] < shape[i]:
            return  // no carry
        index[i] = 0  // carry to next dimension
```

### 8.2 安全性论证

每个 `unsafe` 使用点须有安全性论证：

```markdown
**安全性论证**: `#[repr(C)]` 只用于固定字段顺序与 C 兼容表示，
本示例的安全前提建立在“按已知字段逐个访问”这一点上，而不是依赖“无 padding”假设。
```

### 8.3 性能考量表

```markdown
| 方面 | 设计决策 |
|------|----------|
| 表示开销 | 静态维度类型应保持低开销表示 |
| ZST 优化 | 零维相关类型可利用 Rust 类型系统表达零维语义 |
| 内联 | 仅对确有必要的元数据访问路径说明内联意图 |
| 单态化 | 泛型设计应避免不必要的动态分发开销 |
```

## 9. 实现任务拆分

### 9.1 统一任务粒度

每个任务一般不超过 **10 分钟**，范围限定在"单个函数或单个 trait impl"。

### 9.2 Wave/Phase 分组

按依赖关系分 Wave 或 Phase，同 Wave 内任务可并行：

```markdown
### Wave 1: 基础设施

- [ ] **T1**: 创建 `src/iter/mod.rs` 骨架
  - 文件: `src/iter/mod.rs`
  - 内容: 模块声明、子模块文件占位、公共导出声明
  - 测试: 编译通过
  - 前置: tensor 模块完成
  - 预计: 5 min
```

### 9.3 任务模板（必须）

每个任务必须包含：

| 字段 | 说明 |
|------|------|
| 任务 ID | T1, T2, ... |
| 描述 | 一句话描述 |
| 文件 | 涉及的文件路径 |
| 内容 | 具体实现内容 |
| 测试 | 对应的测试函数 |
| 前置 | 依赖的任务 |
| 预计 | 预计时间 |

### 9.4 并行执行分组图

用 ASCII 图展示 Wave 间的依赖和并行关系：

## 10. 测试计划

### 10.1 测试分类表

```markdown
| 类型 | 位置 | 目的 |
|------|------|------|
| 单元测试 | 适当的测试模块 | 验证单个函数/方法 |
| 集成测试 | 适当的集成测试位置 | 验证跨模块交互 |
| 边界测试 | 与对应测试集合配套组织 | 空数组、单元素、NaN/Inf、非连续 |
| 属性测试（按需） | 适当的测试目录或测试模块 | 对适合抽象不变量的模块，用随机生成补充验证 |
| Feature gate / 配置测试 | 配置矩阵或等效测试位置 | 验证可选 SIMD/并行路径与默认关闭行为 |
| 类型边界 / 编译期测试 | 编译期测试框架或等效机制 | 验证元素类型约束、非法组合与 trait 边界 |
```

### 10.2 单元测试清单

```markdown
| 测试函数 | 测试内容 | 优先级 |
|----------|----------|--------|
| `test_iter_f_contig` | F-order 连续数组：迭代顺序与物理布局一致 | 高 |
| `test_iter_non_contiguous` | 非连续数组（切片后）：步长跳转正确 | 高 |
| `test_iter_empty` | 空数组：立即返回 None | 中 |
```

### 10.3 边界测试场景表

```markdown
| 场景 | 预期行为 |
|------|----------|
| 空数组 `shape=[0, 3]` | `iter()` 立即结束 |
| 单元素 `shape=[1, 1]` | `iter()` 产出 1 项 |
| 非连续切片 | `iter()` 正确处理步长跳转 |
```

### 10.4 属性测试不变量（按需）

仅当模块存在适合用随机输入表达的稳定不变量时，才需要补充本节。

```markdown
| 不变量 | 测试方法 |
|--------|----------|
| `iter().count() == tensor.len()` | 随机形状 [0..10, 0..10, 0..10] |
| `axis_iter(Axis(i)).count() == shape[i]` | 随机形状 |
```

### 10.5 Feature gate / 配置测试

若模块涉及 SIMD、并行或其他可选能力，测试计划须至少覆盖：

```markdown
| 配置 | 验证点 |
|------|--------|
| 默认配置 | 可选能力默认关闭且 API 语义不变 |
| 启用 SIMD | 结果与标量路径满足同一语义约束 |
| 启用并行 | 不出现二次并行，结果与串行路径一致 |
```

### 10.6 类型边界与编译期测试

设计文档须为类型约束和非法组合预留编译期或配置测试：

```markdown
| 场景 | 测试方式 |
|------|----------|
| `bool` 不参与 `sum` / `unique` | 编译期测试或 trait 约束测试 |
| 非法元素类型 | 编译期失败测试或等效约束验证 |
| 越界 feature 组合 | 配置矩阵测试 |
```

## 11. 模块交互设计

### 11.1 接口约定

用表格描述本模块与其他模块之间的接口约定：

```markdown
| 方向 | 对方模块 | 接口/类型 | 约定 |
|------|----------|-----------|------|
| 消费（输入） | `tensor` | `TensorBase<S, D>` | 调用前须保证 shape 合法 |
| 产出（输出） | `math` | `TensorView<'a, A, D>` | 生命周期绑定原始张量 |
```

### 11.2 数据流描述

用 ASCII 图或顺序步骤描述模块间的数据流转：

```
User calls tensor.iter()
    │
    ├── iter/mod.rs constructs Elements<'a, A, D> (holding an immutable reference)
    │       └── yields &A element by element during iteration
    │
    └── math/ module consumes the iterator to perform computation
```

### 11.3 生命周期与所有权约定（按需）

若模块边界涉及生命周期或所有权转移，须明确声明：

```markdown
> **约定**: `iter()` 返回的迭代器借用原始张量，调用期间不得对张量进行可变操作。
> 所有权不转移，迭代器析构后引用自动释放。
```

## 12. 错误处理与语义边界

每个模块设计文档必须明确区分可恢复错误、不可恢复错误（panic）及其诊断信息，并说明不同执行路径下的外部语义是否一致：

```markdown
| 主题 | 需要说明的内容 |
|------|----------------|
| Recoverable error | 返回条件、错误类型、携带的上下文（如轴、shape、索引、目标类型） |
| Panic | 哪些算术域错误或不变量破坏会 panic |
| 路径一致性 | 标量 / SIMD / 并行路径是否保持相同形状、错误类别和数值语义 |
| 容差边界 | 若存在已知舍入差异，误差容差的适用范围与文档位置 |
```

若模块不涉及 SIMD、并行或浮点误差容差，也须明确写出“不适用”，而不是省略。

## 13. 设计决策记录

每个重要设计决策须用四段式 ADR 格式记录：

```markdown
### 决策 N：决策标题

| 属性 | 值 |
|------|-----|
| 决策 | 性能测量采用仓库既有 benchmark 规范 |
| 理由 | 保持测量口径、输入规模和回归阈值一致 |
| 替代方案 | 在模块文档中自定义 benchmark 结构 — 放弃，难以横向比较 |
| 替代方案 | 仅保留定性性能描述 — 放弃，无法支撑回归判断 |
```

或展开式：

```markdown
### 决策 2：广播结果统一视为只读视图

**背景**：广播结果与源张量共享底层数据，但广播会引入零步长布局。

**选项**：
1. 允许广播结果保留原可写性
2. 统一降级为只读/共享只读视图

**选择**：统一视为只读或共享只读视图

**理由**：
- 符合需求说明书对广播语义的约束
- 避免零步长布局下的别名写入风险
- 与转置等共享底层数据的视图语义更一致
- 错误边界更清晰
```

## 14. 性能描述

### 14.1 性能数据（按需）

若模块涉及性能关键路径、SIMD、并行或明确的性能目标，应优先给出可复核的量化指标；
否则可只说明复杂度、内存行为和潜在瓶颈，不强制要求具体测量数字。

以下表格仅作可选示例：

```markdown
| 场景 | 实现路径 | 每元素开销 | 缓存友好性 |
|------|----------|-----------|-----------|
| 连续 F-order | 指针递增 | 1 次指针加法 | 最优（顺序访问） |
| 非连续（切片） | 步长状态机 | 多次乘加运算 | 较差（跳跃访问） |

**性能数据（参考）**:

| 操作 | 连续数组 | 非连续数组（步长=2） | 性能比 |
|------|----------|---------------------|--------|
| 遍历 1M 元素 | ~1ms | ~3ms | 3x |
| 缓存命中率 | ~95% | ~60% | - |
```

### 14.2 复杂度标注

关键操作标注时间/空间复杂度：

```markdown
- 视图创建：O(1)，不分配内存
- transpose：O(1)，仅更新 shape 与 stride 元数据
- contiguous：对已连续 F-order 输入可为 O(1) 复用；对合法非连续输入为 O(n) 复制
```

## 17. 性能与 Benchmark 附加规范（按需）

仅当模块涉及性能关键路径、SIMD、并行或需要量化性能目标时，才需要应用本节要求。

本节用于约束 benchmark 设计文档的写法，不预设具体 benchmark 框架，也不要求引入外部对比对象。

### 17.1 四级分类体系

```
Benchmark 分类
├── Micro-benchmarks        # 单操作、单函数级别
├── Kernel benchmarks       # 核心计算内核
├── Workflow benchmarks     # 真实使用模式
└── Comparison benchmarks   # 与仓库内基线实现或既有路径对比（如串行/并行、标量/SIMD）
```

### 17.2 参数矩阵标准化

若模块需要 benchmark，可从以下参数组合中选择**有意义子集**：

#### 输入规模
| 级别 | 1D | 2D | 3D |
|------|-----|-----|-----|
| **Small** | 64 | 8×8 | 4×4×4 |
| **Medium** | 65,536 | 256×256 | 64×32×32 |
| **Large** | 16,777,216 | 4096×4096 | 256×256×256 |

#### 数据类型
| 类型 | 优先级 | 说明 |
|------|--------|------|
| `f64` | 建议优先 | 科学计算常用精度 |
| `f32` | 建议优先 | 常见于 SIMD 或吞吐敏感场景 |
| `i64` | 按需 | 整数运算路径验证 |
| `i32` | 按需 | 整数运算路径验证 |
| `Complex<f64>` | 按需 | 复数运算路径验证 |
| `Complex<f32>` | 按需 | 复数运算路径验证 |

#### 内存布局
| 布局 | 构造方式 | 验证目标 |
|------|----------|----------|
| F-contiguous | 通过标准构造操作创建的连续拥有型张量 | 默认路径性能基线 |
| Non-contiguous | 由合法切片或转置操作得到的非连续视图 | 非连续路径标量回退惩罚 |

### 17.3 Benchmark 清单表
```markdown
| 组 ID | 操作 | 规模 | 类型 | 布局 | 说明 |
|-------|------|------|------|------|------|
| `elem_add_f64` | `a + b` | S/M/L | f64 | F-contiguous | 连续数组相加 |
| `elem_add_sliced` | `a + b`（b 为切片视图） | M | f64 | Non-contiguous | 非连续惩罚 |
```

### 17.4 CI 三级工作流
若仓库已有 benchmark 执行流程，可参考以下分层方式组织：

| 工作流 | 基准数量 | 预计时间 | 频率 |
|--------|----------|----------|------|
| Smoke Test | 3 个核心文件 × `--quick` | ~5 min | 每次 PR |
| Regression Check | 1-2 个热点基准 | ~10 min | 每次 PR |
| Full Benchmark | 全部文件 × 4 feature 组合 | ~60 min | 每周 |

### 17.5 回归阈值
若模块需要定义性能回归门槛，应由仓库既有 benchmark 流程统一约束；下表仅作分级示例：

| 级别 | 阈值 | 动作 |
|------|------|------|
| **警告** | > 5% 变慢 | CI warning（不阻塞合并） |
| **失败** | > 20% 变慢 | CI failure（阻塞合并） |
| **改善** | > 5% 变快 | CI note（记录性能改善） |
