# 实施方案文档规范

> 本文档定义项目中所有实施方案（implementation plan）的编写规范。
> 实施方案文档是设计方案文档的可执行伴生文档，聚焦于任务的划分、实施顺序和验证方式。
> 每个模块的实施方案必须遵循本规范，确保实施过程可追踪、可验证、可并行。

---

## 1. 文件命名与编号

### 1.1 扁平全局编号

采用 `NN-模块名-impl.md` 的扁平编号格式，与设计方案文档一一对应：

```
02-dimension-impl.md
05-storage-impl.md
07-tensor-impl.md
10-iterator-impl.md
...
```

**规则**：编号与对应的设计方案文档一致，后缀 `-impl` 标识为实施方案。

### 1.2 文件位置

实施方案文档与设计方案文档同目录放置：

```
docs/design/
├── 02-dimension.md            # 设计方案
├── 02-dimension-impl.md       # 对应实施方案
├── 05-storage.md
├── 05-storage-impl.md
└── ...
```

## 2. 文档元信息头

每个实施方案文档开头必须包含元信息块：

```markdown
# <模块名>模块实施方案

> 对应设计: `02-dimension.md` | 模块: `src/dimension/` | 阶段: Phase X
> 前置实施方案: `01-architecture-overview-impl.md`, ...
> 预计总工时: XX min（共 N 个任务）
> 当前状态: 未开始 / 进行中 / 已完成
```

**字段说明**：

| 字段 | 说明 |
|------|------|
| 对应设计 | 关联的设计方案文档编号和文件名 |
| 模块 | 源代码路径 |
| 阶段 | 项目阶段（Phase 1~5） |
| 前置实施方案 | 依赖的其他模块实施方案 |
| 预计总工时 | 所有任务预计时间之和 |
| 当前状态 | 实施方案整体进度 |

## 4. 实施概览

### 4.1 范围声明（必须）

明确列出本次实施包含和不包含的内容：

```markdown
| 范围 | 包含 | 不包含 |
|------|------|--------|
| 基础类型 | Ix0~Ix6, IxDyn, Dimension trait | BroadcastDim（由 broadcast 模块实施） |
| 辅助 trait | IntoDimension, RemoveAxis, Reverse | NdProducer（由 iter 模块实施） |
| 测试 | 单元测试 + 集成测试 + 边界测试 | 属性测试（后续迭代） |
```

### 4.2 实施策略

用简洁表格描述实施策略：

```markdown
| 策略 | 说明 |
|------|------|
| 实施顺序 | 自底向上，先 trait 定义后实现 |
| 并行度 | Wave 内任务可并行，Wave 间串行 |
| 验证频率 | 每个任务完成后立即运行对应测试 |
| 回滚策略 | 每个任务完成后 git commit，失败时可按任务回滚 |
```

### 4.3 实施约束

列出实施过程中的硬性约束：

```markdown
| 约束 | 说明 |
|------|------|
| 编译优先 | 每个任务完成后 `cargo check` 必须通过 |
| 禁止 unwrap | 库代码中使用 `Result`，禁止 `unwrap()` |
| 禁止 as any | 不允许 `as any`、`@ts-ignore` 等类型抑制 |
| 最小改动 | 每个任务仅涉及指定的文件和内容 |
```

## 6. 任务列表

### 6.1 任务粒度标准

| 标准 | 说明 |
|------|------|
| 单一职责 | 每个任务只做一件事 |
| 预计时间 | 每个任务 5~15 分钟 |
| 范围限定 | 涉及 1~3 个文件 |
| 可独立验证 | 完成后可独立运行测试验证 |
| 可独立回滚 | git commit 粒度与任务对齐 |

### 6.2 Wave/Phase 分组规则

- 按**依赖关系**分 Wave，同 Wave 内任务无互相依赖
- Wave 内任务**可并行执行**
- Wave 间按依赖方向**串行执行**
- 每个模块的实施从 Wave 1 开始编号

### 6.3 任务模板（必须）

每个任务必须使用以下模板：

```markdown
### Wave N: <Wave 名称>

- [ ] **T<N>**: <一句话描述>
  - 文件: `<涉及的文件路径>`
  - 内容: <具体实现内容，引用设计方案章节>
  - 测试: <对应的测试函数名或验证方式>
  - 前置: <依赖的任务 ID，或"无">
  - 预计: <N min>
  - 状态: 未开始 / 进行中 / 已完成
  - 实际: <实际耗时，完成后填写>
  - 备注: <补充说明，按需>
```

**字段要求**：

| 字段 | 必须 | 说明 |
|------|------|------|
| 任务 ID | ✅ | T1, T2, ...，全局递增 |
| 描述 | ✅ | 一句话描述做什么 |
| 文件 | ✅ | 涉及的具体文件路径 |
| 内容 | ✅ | 实现的具体内容，引用设计方案对应章节 |
| 测试 | ✅ | 对应测试函数或验证方式 |
| 前置 | ✅ | 依赖的任务 ID |
| 预计 | ✅ | 预估时间 |
| 状态 | ✅ | 任务进度 |
| 实际 | ✅ | 完成后填写实际耗时 |
| 备注 | 按需 | 补充信息 |

### 6.4 任务内容引用设计文档

任务的"内容"字段必须引用设计方案的具体章节，形成可追溯的链接：

```markdown
- [ ] **T3**: 实现 `Ix0` 零维标量
  - 内容: `Ix0` 结构体 + `Dimension` impl，参见 `02-dimension.md §4.2`
```

### 6.5 并行执行分组图（必须）

用 ASCII 图展示 Wave 间的依赖和并行关系：

```
Wave 1: [T1] → [T2]
                      │
           ┌──────────┼──────────┐
           ▼          ▼          ▼
Wave 2:   [T3]       [T4]      [T6]
           │          │
           ▼          ▼
Wave 3:   [T5] ────→ [T7]
                      │
                      ▼
Wave 4:              [T8] → [T9]
```

## 7. 测试实施计划

### 7.1 测试编写策略

| 策略 | 说明 |
|------|------|
| 同步编写 | 每个任务完成后立即编写对应测试 |
| 测试先行 | 复杂逻辑建议先写测试（TDD），再写实现 |
| 测试粒度 | 一个测试函数验证一个行为 |
| 测试命名 | `test_<模块>_<功能>_<场景>` |

### 7.2 测试与任务对应表（必须）

每个任务至少关联一个测试，形成任务-测试对应表：

```markdown
| 任务 ID | 测试函数 | 测试类型 | 优先级 |
|---------|----------|----------|--------|
| T3 | `test_ix0_size_is_one` | 单元测试 | 高 |
| T3 | `test_ix0_ndim_is_zero` | 单元测试 | 高 |
| T4 | `test_ix1_strides_f_order` | 单元测试 | 高 |
| T12 | `test_dimension_integration` | 集成测试 | 高 |
| — | `test_dim_property_invariants` | 属性测试 | 中 |
```

### 7.3 测试分类与执行顺序

| 阶段 | 执行的测试 | 频率 |
|------|-----------|------|
| 任务完成时 | 当前任务的单元测试 | 每次任务完成 |
| Wave 完成时 | Wave 内所有任务的测试 | 每次 Wave 完成 |
| 模块完成时 | 完整测试套件（单元 + 集成 + 边界） | 模块收尾 |
| 集成验证时 | `cargo test` 全量测试 | 关键节点 |

### 7.4 边界测试场景表

从设计方案 §9（测试计划）提取边界场景，形成独立检查表：

```markdown
| 场景 | 关联任务 | 预期行为 | 测试状态 |
|------|----------|----------|----------|
| 空数组 `shape=[0, 3]` | T3 | `iter()` 立即结束 | ☐ |
| 单元素 `shape=[1, 1]` | T4 | `iter()` 产出 1 项 | ☐ |
| 非连续切片 | T5 | 正确处理步长跳转 | ☐ |
```

### 7.5 属性测试计划

列出需通过属性测试验证的不变量：

```markdown
| 不变量 | 测试方法 | 关联任务 | 状态 |
|--------|----------|----------|------|
| `iter().count() == tensor.len()` | 随机形状 `[0..10]^3` | T3 | ☐ |
| `dim.size() == dim.slice().iter().product()` | 随机形状 | T5 | ☐ |
```

## 8. 质量检查清单

### 8.1 任务完成标准

每个任务标记为"已完成"前，必须满足以下条件：

```markdown
| 检查项 | 说明 | 验证方式 |
|--------|------|----------|
| 代码编译 | `cargo check` 无错误 | 命令执行 |
| 测试通过 | 对应测试全部通过 | `cargo test` |
| 格式正确 | `cargo fmt --check` 通过 | 命令执行 |
| 无警告 | `cargo check` 无新警告 | 命令执行 |
| 内容完整 | 实现了任务描述的所有内容 | 代码审查 |
| 范围合规 | 未修改任务范围外的文件 | git diff |
```

### 8.2 Wave 完成标准

| 检查项 | 说明 |
|--------|------|
| Wave 内所有任务已完成 | 每个任务均满足 §8.1 标准 |
| Wave 内所有测试通过 | `cargo test` 全量通过 |
| 无回归 | 新代码未破坏已有测试 |

### 8.3 模块完成标准

| 检查项 | 说明 |
|--------|------|
| 所有 Wave 已完成 | 全部 Wave 满足 §8.2 标准 |
| 完整测试套件通过 | 单元 + 集成 + 边界 + 属性测试 |
| `cargo doc` 无警告 | 文档注释完整 |
| 代码审查通过 | 对照设计方案逐项验证 |
| 设计方案对齐 | 实现与设计方案文档完全一致 |

### 8.4 质量检查清单表格（必须）

每个实施方案文档末尾须包含完整检查清单：

## 9. 实施日志

### 9.1 日志格式

实施方案文档必须包含实施日志，记录实际执行过程：

```markdown
## 6. 实施日志

| 日期 | 任务 | 耗时 | 结果 | 备注 |
|------|------|------|------|------|
| 2026-04-09 | T1 | 5 min | ✅ 完成 | — |
| 2026-04-09 | T2 | 12 min | ✅ 完成 | 比预计多 2 min，因 trait 约束调整 |
| 2026-04-09 | T3 | 8 min | ❌ 失败 | 编译错误：缺少 Sealed impl，已修复 |
| 2026-04-09 | T3 | 10 min | ✅ 完成 | 修复后通过 |
```

### 9.2 日志字段说明

| 字段 | 必须 | 说明 |
|------|------|------|
| 日期 | ✅ | 实际执行日期 |
| 任务 | ✅ | 任务 ID |
| 耗时 | ✅ | 实际耗时 |
| 结果 | ✅ | ✅ 完成 / ❌ 失败 / ⏸ 中断 |

当实际执行与计划有显著偏差时，须记录偏差和调整措施：

```markdown
### 偏差记录
```

## 10. 文档编写通则

### 10.1 语言

- 文档主体使用中文
- 代码、注释、变量名、类型名、命令必须使用英文
- 实施日志中使用简洁、客观的描述

### 10.2 表格优先

与设计方案规范一致，能用表格表达的信息优先用表格。

### 10.3 状态标记

| 标记 | 含义 |
|------|------|
| `- [ ]` | 任务未开始 |
| `- [x]` | 任务已完成 |
| ☐ | 检查项未通过 |
| ☑ | 检查项已通过 |
| ✅ | 成功 |
| ❌ | 失败 |
| ⏸ | 中断/暂停 |

### 10.4 交叉引用

- 引用设计方案时：`参见 02-dimension.md §4.2`
- 引用其他实施方案时：`参见 05-storage-impl.md Wave 2`
- 引用需求文档时：`需求说明书 §3`

### 10.5 版本历史

文档末尾在页脚之前加入版本历史：

```markdown
## 版本历史

| 版本 | 日期 |
|------|------|
| 1.0.0 | 2026-04-09 |
```

### 10.6 页脚

每个文档末尾须包含：

## 11. 与设计方案文档的关系

### 11.1 映射关系

| 设计方案章节 | 实施方案对应 |
|-------------|-------------|
| §6 实现任务拆分（Wave/任务定义） | §3 任务列表（完整展开） |
| §7 测试计划 | §4 测试实施计划 |
| §2 文件位置 | §3 任务列表中每个任务的"文件"字段 |
| §4 公共 API 设计 | §3 任务列表中每个任务的"内容"字段引用 |

### 11.2 职责划分

| 方面 | 设计方案 | 实施方案 |
|------|----------|----------|
| 关注点 | **做什么**、**为什么** | **怎么做**、**何时做** |
| 任务描述 | 粗粒度概述 | 细粒度可执行步骤 |
| 测试描述 | 测试策略和清单 | 测试函数与任务的精确对应 |
| 进度跟踪 | 不涉及 | 实施日志、状态标记 |
| 时间估算 | 概要估算 | 逐任务估算 + 实际记录 |

### 11.3 一致性要求

实施方案的任务内容必须与设计方案的任务拆分保持一致。若实施中发现需要调整：

1. 先更新设计方案文档
2. 再更新实施方案文档
3. 在偏差记录中说明调整原因
